# SPRINT-007 — Source-Layout Single-Slot Decode Correctness (CLAUDE draft)

**Status:** DRAFT 2026-05-17
**Predecessor:** SPRINT-006 (sidecar V100 execution context, descriptor policy, HC relay, no-math layer skeleton)
**Successor:** Prefill, KV, and compressed attention; gated on this sprint shipping at minimum a registered source-dtype primitive set, a guarded diagnostic decode entry point, and a documented correctness verdict against the trusted reference.

---

## Overview

Sprint 006 shipped the V100 execution skeleton: an 8-stage sidecar context, a
descriptor policy that classifies every source/runtime/kernel-family triple,
fail-closed topology and memory checks, an HC relay primitive, and a no-math
layer walk. `model_uses_v100_source_layout` recognizes the DSv4-Flash source
GGUF, `weights_validate_v100_source_layout` accepts it, but `ds4_engine_open`
still rejects normal generation by design because no FP8/MXFP4 execution
kernels are wired into the runtime.

Sprint 007 is the correctness gate that has to come before any prefill, KV
growth, or deployment work. Its narrowest possible useful outcome is a set of
**source-dtype scalar/row primitives** that can either compute exact reference
matvecs over BF16, F32, F8_E4M3_B128, and MXFP4 source bytes, or fail closed
with a precise unsupported-format error. Its broadest useful outcome is a
**guarded single-slot, small-context oracle decode path** that compares
generated tokens (or top-logprob slices) against `tests/test-vectors/official.vec`
on the V100 pod for at least one short prompt.

Sprint 007 does not unlock normal generation. The source-layout guard in
`ds4_engine_open` stays in place. The oracle path is reached only through a
new, named diagnostic entry point that callers have to opt in to and that is
not exposed through any server, CLI, or production decode wiring.

V100 precision policy is unchanged from Sprints 005-006:

- BF16 source bytes are diagnostic/probe input or are converted to FP16/F32 at
  the kernel boundary; there is no native BF16 tensor-core path.
- F8_E4M3_B128 is a packed source/runtime input to a registered unpack/dequant
  kernel; later production kernels feed FP16 HMMA or registered low-bit math.
- MXFP4 is a packed source/runtime input to a registered grouped-low-bit kernel
  or to a bounded unpack-to-FP16 tile path.
- FP16 HMMA with FP32 accumulation is the dense production target.
- FP32 is the oracle math for this sprint's reference path and stays a
  control/reduction/diagnostic format in production; it is not a broad GEMM
  fallback.
- The reference path may be CPU-only and slow; this sprint does not optimize it.

The hard correctness risk that drives the whole sprint is that
F8_E4M3_B128 and MXFP4 source semantics must be locked in exactly. The
project already has a CPU dequant implementation that consumes the HF source
layout in `gguf-tools/deepseek4-quantize.c::dequant_fp8_weight` and
`::dequant_fp4_weight`; the sprint must verify (or revise) that those exact
semantics also apply to the runtime pack rows that
`docs/sprints/drafts/SPRINT-003-PACK-INDEX.tsv` indexes, before any oracle
matvec is trusted.

## Outcome Contract

- **SHIP** if: source-dtype primitives exist for BF16, F32, F8_E4M3_B128, and
  MXFP4 with model-less synthetic tests; the mislabeled F16 paths over
  source BF16 tensors are corrected; the diagnostic guard-unlock entry point
  exists and is wired only to the oracle path; an oracle single-slot decode of
  the first token over `tests/test-vectors/prompts/short_italian_fact.txt` (or
  any other short fixture) runs on the V100 pod with the persistent pack;
  generated bytes or top-logprob membership match the corresponding entry in
  `tests/test-vectors/official.vec` within a documented tolerance; normal
  source-model generation still fails closed; cluster artifacts are archived.
- **EXTEND** if: BF16/F32 primitives plus the diagnostic-mode plumbing land,
  and at least one of F8_E4M3_B128 or MXFP4 primitives lands with model-less
  tests, but oracle decode against the cluster pack is blocked because the
  other format's semantics are not locked in, the pod is unavailable, or one
  family's per-tensor reference fails to match. The exact missing primitive
  and the blocker are recorded, and the sprint ships the completed primitives.
- **STOP** if: F8_E4M3_B128 or MXFP4 source semantics cannot be established
  with sufficient confidence (e.g., the runtime pack rows disagree byte-for-
  byte with the HF dequant in `gguf-tools/deepseek4-quantize.c` and no
  documented variant explains the disagreement); the guard-unlock mechanism
  cannot be made narrow enough to avoid leaking into normal generation; or the
  oracle path drifts into prefill, KV growth, multi-slot scheduling, or
  routed-expert performance work.

## Architecture

### Source Of Truth

`docs/architecture/DS4-V100-LAYOUT.md` remains the architecture anchor. The
oracle path consumes its source-dtype and tensor-family tables verbatim; it
does not invent alternate dispatches. The dequant byte semantics for
F8_E4M3_B128 and MXFP4 must agree byte-for-byte with the HF tooling at
`gguf-tools/deepseek4-quantize.c` lines 681-737. If a disagreement appears,
that is a STOP that escalates to a layout and packer review, not a license to
silently reinterpret bytes.

`docs/sprints/drafts/SPRINT-003-PACK-INDEX.tsv` is the source-dtype anchor for
the runtime pack rows. The new primitives must read from descriptors built off
that index (already validated by Sprint 006's descriptor binding) rather than
re-parsing GGUF bytes.

The new oracle module is a **sidecar** to the existing CPU reference paths
in `ds4.c` (lines 6201, 7971, 8093 for `output_logits_one`; line 8037 for
`forward_first_token_cpu`). It does not delete the legacy-layout
`matvec_q8_0`/`matvec_f16` paths; it dispatches by descriptor over the source
layout. Legacy-layout generation keeps its current code path.

### Source-Dtype Primitives

The minimal primitive set, all FP32-internal:

```c
/* BF16: bit-exact 16-bit → 32-bit expand. Already exists in
 * gguf-tools/quants.c as bf16_to_f32_bits — promote to shared header. */
void ds4_v100_oracle_bf16_row_to_f32(float *out,
                                     const uint16_t *src,
                                     uint64_t n_elements);

/* F32: source-faithful copy with stride accounting (no dequant). */
void ds4_v100_oracle_f32_row_copy(float *out,
                                  const float *src,
                                  uint64_t n_elements);

/* F8_E4M3_B128: dequant a row using the 128-block E8M0 scale layout
 * documented in gguf-tools/deepseek4-quantize.c::dequant_fp8_weight.
 * Scale offset and stride come from the descriptor. */
int ds4_v100_oracle_f8_e4m3_b128_row_to_f32(float *out,
                                            const uint8_t *src_row,
                                            const uint8_t *scale_row,
                                            uint64_t in_dim,
                                            char *err, size_t errlen);

/* MXFP4: dequant a row using the 32-block E8M0 scale layout and the
 * 16-entry FP4 codebook documented in
 * gguf-tools/deepseek4-quantize.c::dequant_fp4_weight. */
int ds4_v100_oracle_mxfp4_row_to_f32(float *out,
                                     const uint8_t *src_row,
                                     const uint8_t *scale_row,
                                     uint64_t in_dim,
                                     char *err, size_t errlen);
```

Every primitive returns nonzero on bad shape, bad scale stride, scale-row
size mismatch, or unsupported variant. The classifier from Sprint 006
(`ds4_v100_classify_or_die`) is the source-of-truth for which descriptors are
eligible to call which primitive.

### Per-Family Oracle Matvec Drivers

The oracle decode path needs four matvec families (FP32 internal):

| Driver | Descriptor source-dtype + family | Notes |
|---|---|---|
| `ds4_v100_oracle_matvec_bf16` | BF16 / `BF16_GLOBAL` | Token embedding row gather + output head BF16 → FP32 matvec |
| `ds4_v100_oracle_matvec_f32` | F32 / `F32_CONTROL` and `HC_CONTROL` | HC pre/post, attention/ffn norms, router scores, output norm, attn sinks |
| `ds4_v100_oracle_matvec_f8_e4m3_b128` | F8_E4M3_B128 / `FP8_DENSE` | attn_q_a, attn_q_b, attn_kv_latent, attn_output_a, attn_output_b, indexer_attn_q_b, shared FFN gate/up/down |
| `ds4_v100_oracle_matvec_mxfp4` | MXFP4 / `MXFP4_EXPERT` | routed FFN gate_exps, up_exps, down_exps — driver iterates only the active expert rows selected by the router |

Each driver:

- takes a `ds4_v100_tensor_descriptor` (semantic id, owning GPU, source dtype,
  arena offset, byte length, scale offset, rows/cols, layer id);
- reads bytes via a host-side pull from the resident pack (Sprint 004 GGUF or
  shard provider; not Sprint 006 device arenas in this sprint — see Open
  Question 1);
- runs the matvec into an FP32 accumulator;
- never materializes a persistent dequantized weight copy beyond the row tile
  in scratch;
- refuses unknown `(source_dtype, family)` pairings via the Sprint 006
  classifier.

The drivers are reference, not production. They are slow by design. The
production kernels (FP8 unpack + FP16 HMMA tile, MXFP4 grouped low-bit, BF16
→ FP16 conversion stub for the head) remain deferred to a later sprint.

### Mislabeled-Reference Cleanup

The current CPU reference paths in `ds4.c` were written for the legacy q2/q4
DS4 GGUF family and reach for `matvec_f16`/`matvec_q8_0` regardless of source
dtype. For the source layout, the following are mislabeled:

- `embed_token_f16` is called with a BF16 `token_embd.weight`; the existing
  body already handles BF16 internally but the symbol name and dispatch
  surface are misleading.
- `matvec_f16(pre, model, weights->output_hc_fn, flat)` in `output_hc_head_one`
  reads an F32 tensor (`output_hc_fn`) as F16.
- `matvec_q8_0(logits, model, weights->output, norm)` in `output_logits_one`
  reads a BF16 source `output.weight` as Q8_0.
- The same `matvec_q8_0` pattern is used in attention/shared-expert
  projections that for the source layout are F8_E4M3_B128, not Q8_0.

The cleanup is conservative: introduce
`ds4_v100_oracle_matvec_dispatch(descriptor, x, out)` and route the source
layout through it. Legacy-layout call sites in `ds4.c` keep their existing
`matvec_q8_0`/`matvec_f16` calls under an `if (!w->v100_source_layout)`
branch. The oracle path takes the source-layout branch and never falls back
to the legacy matvec.

### Guarded Diagnostic Entry Point

The source-layout generation guard in `ds4.c::ds4_engine_open` (around line
17305) is preserved verbatim. A new option is added to
`ds4_engine_options`:

```c
/* Opaque diagnostic unlock token. The only accepted value is the
 * literal string in ds4_v100_oracle.h:
 *   #define DS4_V100_ORACLE_UNLOCK_TOKEN "ds4-v100-oracle-decode-one-slot"
 * Any other non-NULL value fails closed with the same message as a
 * missing token. */
const char *v100_oracle_unlock_token;
```

The engine-open branch is:

```c
if (e->weights.v100_source_layout && !opt->inspect_only) {
    if (opt->v100_oracle_unlock_token &&
        strcmp(opt->v100_oracle_unlock_token,
               DS4_V100_ORACLE_UNLOCK_TOKEN) == 0) {
        e->oracle_only = true;  /* normal generate() stays disabled */
    } else {
        fprintf(stderr, "ds4: native DS4-Flash source layout ...\n");
        return 1;  /* unchanged guard */
    }
}
```

`e->oracle_only` permits only the new `ds4_v100_oracle_decode_one_token(...)`
entry point. Normal `ds4_generate`, `ds4_decode`, `ds4_server` request
handling, and the eval/bench/CLI wrappers either do not see `oracle_only`
engines (they fail closed at engine-open if the unlock token is not provided)
or check `oracle_only` at the start and refuse with a specific message.

The unlock token is **not** an environment variable. Putting it on the
options struct keeps it out of CI and out of `--help` output; the tool that
needs it sets it in C code. The token string is not a secret; the value of
naming it is that an audit can grep for `DS4_V100_ORACLE_UNLOCK_TOKEN` and
find every site that opts into oracle mode.

### Oracle Single-Slot Decode Path

```c
typedef struct {
    int             token_id;
    float           top_logprob;
    int             top_k;
    int            *top_token_ids;     /* size top_k */
    float          *top_logprobs;      /* size top_k */
} ds4_v100_oracle_decode_result;

int ds4_v100_oracle_decode_one_token(ds4_engine *engine,
                                     const int *prompt_token_ids,
                                     uint32_t   n_prompt_tokens,
                                     uint32_t   top_k,
                                     ds4_v100_oracle_decode_result *out,
                                     char *err, size_t errlen);
```

Behavior:

- Uses the existing CPU forward graph (`forward_first_token_cpu` in `ds4.c`)
  as a starting scaffold but dispatches matvecs through
  `ds4_v100_oracle_matvec_dispatch` for source-layout descriptors.
- Single slot, single token. No KV growth beyond what the existing CPU graph
  already allocates as scratch for one slot. No prefill rewind.
- Top-K logprobs are computed by sorting the full logits row and returning
  the K largest with their token ids. K is bounded (default 20 to match the
  official-API capture).
- Runs entirely on CPU. The Sprint 006 CUDA context is not required for the
  oracle path in this sprint; descriptors are consulted for source-dtype and
  arena-offset metadata, but byte reads go through the existing pack/GGUF
  providers, not through `cudaMemcpy`. (See Open Question 1.)
- Fails closed on any unsupported tensor family pairing surfaced by the
  classifier.

The path is single-slot and FP32-internal; it is not a production decode and
is not subject to throughput requirements. It is allowed to take minutes per
token on the cluster.

### Reference Comparison

`official.vec` already encodes prompt + expected continuation token bytes. The
sprint adds a minimal driver next to `ds4_test --logprob-vectors`:

```sh
./tools/ds4-v100-oracle-decode \
    --model /models/DSv4-Flash-256e-fixed.gguf \
    --pack-index docs/sprints/drafts/SPRINT-003-PACK-INDEX.tsv \
    --prompt tests/test-vectors/prompts/short_italian_fact.txt \
    --official tests/test-vectors/official.vec \
    --case short_italian_fact \
    --top-k 20 \
    --json /tmp/oracle.json
```

Outputs:

- the first generated token id and decoded bytes;
- top-20 token ids with logprobs;
- pass/fail verdict against the official-vector entry:
  - **exact match** if the first generated token id equals the official first
    token id;
  - **top-K membership** match if the official first token id appears in the
    local top-K with logprob within the configured tolerance;
  - **fail** otherwise (with diff details).

The verdict format must be machine-readable so the cluster log can be parsed
by the sprint report.

The legacy `ds4_test --logprob-vectors` runner is not modified in this
sprint; the oracle path lives in a separate tool because it requires the
unlock-token option and runs on the V100 pod, not the laptop.

## Security

- The unlock token is a code-level constant, not a runtime secret. Its
  purpose is auditability (grep for `DS4_V100_ORACLE_UNLOCK_TOKEN`), not
  authentication.
- The oracle path is not exposed through `ds4-server`. The server entry
  points do not set the unlock token and refuse oracle-only engines.
- The CLI tool `ds4-v100-oracle-decode` reads the prompt file, the pack
  index, the official vector file, and the model. All paths are validated:
  no path traversal into the source model, the pack index, or any shard
  file; the JSON output file is opened with `O_CREAT | O_EXCL` to avoid
  overwriting prior artifacts.
- All matvec call sites validate `descriptor->byte_length`,
  `descriptor->rows`, `descriptor->cols`, `descriptor->arena_offset`,
  `descriptor->scale_offset`, and `in_dim` before any byte read. Integer
  overflow in `rows * cols * sizeof(float)` and
  `scale_row_bytes = in_dim / block` is checked with overflow-safe math.
- The Sprint 006 source-layout generation guard text is preserved verbatim
  so log scrapers and CI tests that match on it keep working.
- Model bytes are treated as read-only; no oracle path writes back into the
  pack, the source GGUF, or any shard.
- The oracle's top-K computation bounds K (default 20, max 1024) and
  validates that the requested K is no larger than `DS4_N_VOCAB`.
- The synthetic-fixture guard test does not depend on a real source model
  on disk; it constructs the minimum necessary in-memory model structure to
  trigger `model_uses_v100_source_layout`, so the laptop test does not need
  the 145 GiB model.

## Open Questions

1. **Do oracle matvecs pull bytes from the resident `ds4_gpu_arena` or from
   the host-side pack/GGUF providers?** This draft proposes **host-side**
   reads (via the existing pack providers) for Sprint 007 to keep the
   oracle CPU-only and avoid pulling CUDA scheduling into the
   correctness-gate sprint. The Sprint 006 device arenas remain available
   for a future sprint to upgrade the oracle to device-side once
   correctness is established. If the host-side path is too slow on the
   cluster, Phase 5 EXTENDs with the timing recorded and Sprint 008 picks
   up the device-side oracle as its first task.

2. **Is top-K logprob membership sufficient as a pass criterion, or must
   the local first-token id exactly match the official first-token id?**
   This draft proposes **exact first-token id match is the SHIP bar**, with
   top-K membership accepted as an EXTEND-grade signal (correctness is
   plausible but not confirmed). The official vectors store the API's
   greedy continuation, so an exact match is achievable when the oracle is
   correct. Tolerance for top-K logprob value comparison is set generously
   (e.g., 1e-2 in logprob space) because the official API does not expose
   full logits.

3. **Per-family or whole-graph oracle?** This draft proposes the oracle
   wraps the existing `forward_first_token_cpu` orchestration with a
   descriptor-keyed dispatch installed at every matvec call site. The
   per-family drivers are individually testable via the dispatch smoke and
   collectively testable via the synthetic decode. This avoids the
   complexity of writing a second forward graph while keeping per-family
   failure modes diagnosable.

4. **Should the unlock token be a string constant or a build-time `-D`
   define?** This draft proposes a **string constant** on the options
   struct. A build-time define would prevent the same binary from running
   both oracle and legacy paths, which makes regression testing harder; the
   string constant approach is auditable by grep, never logged in
   `--help`, and never set by any non-oracle entry point.

5. **What cluster time budget is acceptable for one oracle token?** This
   draft proposes a **30-minute soft cap per token** for the V100 pod. If
   the first token takes longer, Phase 5 still archives the partial result
   and EXTENDs. The next sprint can address speed (device-side oracle, FP8
   unpack on V100, etc.) once correctness is verified at any speed.

Remaining merge-time questions:

- Whether to capture a per-prompt JSON `oracle.json` artifact in
  `docs/sprints/drafts/SPRINT-007-ORACLE-*.json` for diff against future
  runs (probably yes, but it is large and may be archived only as a SHA).
- Whether to add a `--reference-json` flag that takes a Sprint 007-captured
  local JSON as a golden, so subsequent sprints can regression-test

- Whether the F32 control primitive should also accept the I32 tensors
  used by the hash-table rows (layers 0-2 `ffn_gate_tid2eid`), or whether

  F32_CONTROL family in the dispatch, since the existing classifier
  already routes it there.

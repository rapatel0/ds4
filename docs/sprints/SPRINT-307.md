# Sprint 307 - TP/EP Reference Parity Harness

Date: 2026-05-24

## Goal

Create an end-to-end parity harness that compares the tokenizer-enabled TP/EP
server against official DS4 selected-token vectors.

## Implementation

- Added `tools/ds4-v100-tp-ep-reference-parity.py`.
- The harness parses `tests/test-vectors/official.vec`.
- It drives the live TP/EP HTTP API, not internal helpers, so the gate covers:
  tokenizer input, prompt prefill, decode, output-head selection,
  detokenization, response metadata, and resident cursor movement.
- It compares decoded generated bytes against the official selected-token byte
  sequence for each vector case.

## Interpretation

This is the first explicit production-readiness gate, and it fails. That is
the correct current signal: the TP/EP appliance is operational as a diagnostic
text server, but generated tokens are not yet faithful to the official DS4
reference path.

The next work must focus on semantic parity, especially the bridge HC sequence,
attention/FFN ordering, and final output-head inputs. More API wrappers or
throughput tuning should not be treated as production progress until this gate
passes.
